# 项目二：金融研报问答 —— RAG 系统

## 为什么做 RAG，而且选金融场景

RAG 是目前面试被问到概率最高的模块，没有之一。字节、阿里、腾讯、美团，几乎每场都会问。

但这个方向有一个尴尬的问题：90% 的人做出来都是一个 demo，随便切个 txt 文件，用 LangChain 套一下就完了。面试官一问就知道深浅。

所以关键不在于“要不要做 RAG”，而在于**场景选什么**。

这里选择的是金融研报，原因很简单：PDF 比 txt 难处理太多。研报里全是多栏排版、表格、图表、脚注，不能直接粗暴 split。表格数据全是乱的。面试你只要说“我处理过 PDF 多栏排版和表格切块”，就比纯文本选手听起来更有东西。

另外金融问答有天然优势：答案是确定的。

比如：

> 比亚迪 Q3 毛利率多少？

研报白纸黑字写着，评测非常好做。不像开放问答那样评测全是主观判断。

再加上字节、蚂蚁、百度、阿里都有金融 AI 团队，面试官对这个场景很熟，不用花时间解释业务背景。

具体任务就是：

> 给分析师做一套研报问答工具。自然语言提问，系统自动从几百份 PDF 里找到相关段落，生成回答，并标注来源文档和页码。

---

## 数据去哪里找

研报数据其实很好搞。

东方财富 Choice、上市公司年报季报 PDF 可以免费下载，选 3-5 个行业、每个行业 10 份就够了。发改委、民俗行业政策文件也是公开的。如果懒得一个个找，卷心菜研究、墨宝研报都有免费的行业报告可以批量下载。

评测数据可以用 FinanceIQ 或 FinQA，GitHub 和 HuggingFace 都能找到。

总量控制在 **200-300 份 PDF** 就行，太少没有多样性，太多你处理不过来。

面试时别说“下载了一批 PDF”。可以换个说法：

> 语料库 300+ 份金融研报和政策文件，覆盖新能源、半导体、消费三个行业。文档以 PDF 为主，包含多栏排版、表格、图表，部分超过 80 页。评测自建了 120 条，分事实型、对比型、汇总型三类。

按这么说，面试官一般不会再追问数据来源。

## 代码别从零写，先读这三个

项目一的时候说过，从零写是浪费时间。RAG 这里更是如此，一个 PDF 解析就能折腾你一周。

### RAGFlow

仓库：<https://github.com/infiniflow/ragflow>

这是 RAG 方向最值得读的项目。它的 DeepDoc 模块做了深度 PDF 解析，包括表格识别、多栏处理、图表提取等。混合召回链路和评测体系也做得很完整。

建议花差不多三天，把它的切块策略和检索融合部分代码读一遍，收获很大。

### Langchain-Chatchat

仓库：<https://github.com/chatchat-space/Langchain-Chatchat>

中文 RAG 的老牌项目。中文分词切块、FAISS / Milvus 多向量库适配、BM25 + KNN 混合检索这些实现写得比较清晰，适合学习。

### QAnything

仓库：<https://github.com/netease-youdao/QAnything>

网易有道做的项目。它的 2-stage retrieval，也就是先 embedding 粗筛，再 rerank 精排，以及父子文档召回策略比较有意思，rerank 阈值怎么设计也值得看。

我的做法：

> 先花两三天读 RAGFlow 的 DeepDoc 和检索融合代码，理解思路；然后自己用 FAISS + FlagEmbedding 写一套。面试时说“我参考了 RAGFlow 的文档解析思路，但检索和评测链路是自己实现的”，这个说法比较安全。

## 具体怎么做：分四周

## 第 1 周：数据处理 + 基线搭建

### 1. 数据准备：金融研报场景

- 从东方财富 / 尖果等平台下载 **200-300 份 PDF 研报**。
- 选 **3 个行业** 就差不多，例如新能源、半导体、消费。
- PDF 解析建议：
  - 单一难点：用 **PyMuPDF** 提取文本。
  - 表格处理：用 **pdfplumber** 解析表格。
  - 处理页眉、页脚。
- 这个点很容易出彩：
  - 说明你讲清楚了“纯文本 split 在 PDF 上为什么不行”。
  - 表格会被拆散。
  - 多栏内容会串行。
  - 跨页后 29 的对手可能会漏。
- 参考 RAGFlow 的 DeepDoc 模块，学习它的版面分析思路。

### 2. 切块实验

这一步非常重要，很多人都不会认真做，面试基本上会问。

做三组对比：

| chunk size | overlap |
|---:|---:|
| 256 | 50 |
| 512 | 100 |
| 1024 | 200 |

同时记录：

- 召回条数
- 平均长度
- 后续召回的 Recall@5 变化

金融文档有一个特殊点：**表格行不能被切断**。

因此需要实现“表格保护”切块策略。这是一个很好的面试话题。

最终输出：

> 一张切块策略对比表。

### 3. 向量索引构建

- 使用 **FlagEmbedding 的 bge-large-zh-v1.5** 做 embedding。
- 使用 **FAISS** 构建索引：
  - Flat
  - IVF
  - HNSW
- 输出内容：
  - 索引构建脚本
  - 向量维度
  - 文档记录

## 第 2 周：多路召回 + 召回评测

### 1. 实现三路召回

| 路线 | 方法 |
|---|---|
| 路线 A | 纯向量召回（FAISS） |
| 路线 B | 纯 BM25 召回 |
| 路线 C | 混合召回：向量 + BM25 分数加权融合 |

### 2. 构造评测集

金融场景的核心优势是：**答案确定性强**。

自己写 **80-120 条 query、ground_truth_doc_id、answer** 三元组。

问题类型建议覆盖：

- 事实型：例如“比亚迪 Q3 营收多少？”
- 对比型：例如“宁德时代 vs 比亚迪毛利率哪个高？”
- 汇总型：例如“新能源汽车行业 2025 主要政策变化？”

这个类别以后，面试时你就可以说：

> 我的评测集覆盖三类问题，其中对比型和汇总型是纯向量召回的弱点，这也自然引出了为什么要做混合召回。

## 第 3 周：重排 + 生成 + 故障处理

### 1. 接入 Reranker

- 使用 FlagEmbedding 的 **bge-reranker-v2-m3**。
- 对比方案：
  - 向量召回 Top20 → Rerank 取 Top5
  - 直接 Top5
- 记录：
  - Recall
  - 和准确率变化

### 2. 生成约束

为了减少幻觉，需要加入生成约束：

- 要求：只引用检索到的证据回答。
- 如果召回片段不足或不相关，则返回：

> 我不确定，而不是硬编。

可以在 prompt 中加入类似规则：

> 你必须引用给定证据回答。如果证据不足，请明确说明无法从现有材料中确定。

这句话面试百分百会问。

### 3. 故障处理

这一块面试问得很频繁，没有案例会很被动。

| 故障 | 你的 badcase（金融场景具体例子） | 根因 | 怎么修的 |
|---|---|---|---|
| 空召回 | “宁德时代 2025 年装机量”，但文档里写的是“装车量” | 表述差异导致 embedding 语义不够 | 加 query 改写、同义词扩展 |
| 答非所问 | 问“比亚迪毛利率”，召回了营收、股价不相关列表 | chunk 太大，宽泛报表被切成一个块 | 缩小 chunk、表格单独切块、重排 |
| 幻觉 | 模型编了一个不存在的数字 | 生成约束不足 | 加引用约束、后校验 |
| 延迟高 | 响应超过 3 秒 | 重排模型大，PDF 过长导致块多 | 先粗筛再精排、缓存热查询 |

## 面试高频追问

RAG 项目被追问的概率比训练项目还高，这几个问题一定会遇到：

### 1. chunk size 怎么选的？

回答重点：

- 做 256 / 512 / 1024 三组对比。
- 最好记住具体 Recall 数字。

### 2. 为什么不只用向量检索？

可以举例：

> 用户搜“宁德时代装机量”，文档里写的是“装车量”。纯向量可能召不回来，BM25 反而能匹配到关键词。因此混合召回效果更稳。

### 3. Reranker 到底有多大提升？

不要空泛说“提升明显”，要直接报数字。

如果不做实验，你很难回答这个问题，所以实验必须跑。

### 4. 空召回怎么处理？

可回答：

- query 改写
- 同义词扩展
- 降级回答

### 5. 幻觉怎么缓解？

可回答：

- 证据绑定
- 低置信度拒答
- 生成后引用校验

### 6. QPS 多少，瓶颈在哪？

常见瓶颈：

- 重排模型延迟
- 生成模型延迟

讲真，RAG 的追问深度比训练项目大。但好处是，只要每个问题都能用实验数据回答，所以实验做扎实反而最好讲。
