# 金融研报助手 Todo

本文档根据 `docs/project_guide_project2.md` 整理，用来跟踪 `financial_research_assistant` 项目的实际开发计划。

项目目标不是做一个套壳 Demo，而是做出一条可解释、可评测、可演示的金融研报 RAG 链路：PDF 解析 -> 切块 -> 索引 -> 多路召回 -> 重排 -> 生成 -> 引用 -> 评测 -> badcase 复盘。

## 总体策略

- [ ] 先完成最小可运行闭环：少量 PDF、基础解析、基础切块、FAISS 检索、回答和引用展示。
- [ ] 再补齐实验：切块对比、BM25/向量/混合召回对比、FAISS 索引对比、rerank 对比。
- [ ] 最后整理可展示材料：Demo、实验表、系统设计图、badcase 池、面试问答笔记。
- [ ] 本地优先在 MacBook Air 上开发；大模型生成或大 reranker 慢时，先用小模型或 API 替代。

## 阶段 0：项目基础设施

- [ ] 明确项目配置入口，整理 `configs/rag.yaml` 的字段和默认值。
- [ ] 建立基础 Python 包结构，规划核心模块：
  - [ ] `pdf_parser`：PDF 文本、表格和页码解析。
  - [ ] `chunker`：普通切块和表格保护切块。
  - [ ] `retriever`：FAISS、BM25、混合召回。
  - [ ] `reranker`：cross-encoder / FlagEmbedding rerank。
  - [ ] `generator`：基于证据的答案生成。
  - [ ] `evaluator`：Recall、引用命中率、延迟等评测。
- [ ] 设计数据文件格式：
  - [ ] 文档元数据：`doc_id`、标题、行业、来源、年份、页数。
  - [ ] chunk 数据：`chunk_id`、`doc_id`、页码、文本、类型、表格标记。
  - [ ] 评测数据：`query`、`ground_truth_doc_id`、`answer`、问题类型。
- [ ] 补充开发说明文档，包括环境安装、数据放置位置、运行顺序。

## 阶段 6：故障处理与 Badcase 池

目标：把项目做成可复盘、可解释、可面试展开的系统。

- [ ] 建立 badcase 记录格式：
  - [ ] 问题。
  - [ ] 错误现象。
  - [ ] 召回结果。
  - [ ] 根因。
  - [ ] 修复方案。
  - [ ] 修复后效果。
- [ ] 至少积累 10 个典型 badcase。
- [ ] 覆盖以下类型：
  - [ ] 空召回：同义词、简称、行业术语不一致。
  - [ ] 答非所问：chunk 太大或召回主题过宽。
  - [ ] 幻觉：模型编造不存在的数字。
  - [ ] 延迟高：rerank 或生成模型耗时过长。
  - [ ] 引用错误：答案和来源页码不匹配。
- [ ] 实现或记录修复手段：
  - [ ] query 改写。
  - [ ] 同义词扩展。
  - [ ] 缩小 chunk。
  - [ ] 表格单独切块。
  - [ ] rerank。
  - [ ] 证据约束和引用校验。
